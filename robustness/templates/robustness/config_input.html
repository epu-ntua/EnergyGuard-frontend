Config requirements

  • Set a top-level name: and optional description:.
  • Provide dataset details under dataset: — path, format, and target_column.
  • Declare the model under model: with backend, task_type, and MLflow connection fields.
  • List attack scenarios under robustness.attacks: — each with type, epsilon, and attack-specific parameters.
  • Define output paths under output: — output_dir, metrics_filename, and report_filename.

Config field reference

Every field your config.yaml can define. The field name's color shows how important it is.

Required Important Optional
Field Options / template Description
name <run_name> Human-readable run identifier shown in the UI and output files.
description <description> Short note describing the purpose of the run or config.
dataset.path <dataset_uri_or_path> Source dataset used to load the evaluation records.
dataset.target_columns [<target_1>, <target_2>, ...] Columns treated as prediction targets; for multi-target regression, check how per-target loss is aggregated.
dataset.feature_columns [<feature_1>, <feature_2>, ...] Input features passed to the model and attacks; mask out categorical/binary columns if they shouldn't be perturbed.
model.backend mlflow | pytorch | sklearn Selects the loader path used to load the model.
model.task_type classification | regression Sets whether the evaluation is treated as classification or regression.
model.mlflow_model_uri runs:/<run_id>/model or models:/<name>/<version> Points to the MLflow model artifact that will be loaded for evaluation.
model.mlflow_validate_signature true | false Validates the model's input schema; only disable once you've manually verified it matches.
model.mlflow_include_training_metrics true | false Includes original training metrics in the generated report.
model.clip_values [<min>, <max>] Valid input range for gradient-based attacks; must match the actual scale of every feature column.
robustness.attacks [{type: fgsm, ...}, {type: pgd, ...}, {type: bim, ...}, ...] List of attacks to run (fgsm, pgd, bim, ...), each with its own configuration.
robustness.sample_size <positive_integer> Limits how many samples are evaluated; leave it out to use all rows.
attacks.epsilon <epsilon> Perturbation budget, relative to the clip_values range.
attacks.eps_step <step> Per-iteration step size for iterative attacks like PGD and BIM.
attacks.max_iter <positive_integer> Iteration count for iterative attacks like PGD and BIM.
attacks.norm inf | 1 | 2 Distance metric (L∞, L1, L2) bounding the size of the perturbation.
attacks.targeted true | false Whether the attack targets a specific output or just maximizes error generally.
attacks.regression_objective maximize_loss | maximize_deviation Controls what the attack optimizes for in regression tasks.
attacks.feature_mask [<column_name>, ...] Restricts perturbation to specific columns; confirm your library supports this before using mixed-type feature sets.
